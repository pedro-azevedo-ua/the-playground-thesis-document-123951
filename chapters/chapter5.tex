With user requirements, the prototypes can be build.
In this section it will be presented the initial design of the features that matter the most for the users, the navigation flow and the mockups.

The following are some requirements that were design.


\subsection{Courts Module}
\label{subsec:rp-courts}
Following the requirements gathering and persona development phase, the design of the Courts module was focused on two primary user goals: finding a suitable court and assessing its real-time conditions. The design is structured into two main stages: discovery and interaction.

\subsubsection{Court Discovery and Selection}

The discovery phase offers two primary interfaces for court selection. The first is a list-based browser (Figure \ref{fig:courtsListMU}), where users can scroll through courts sorted by customizable criteria. Each "court card" provides essential metadata, such as the number of courts and hoops, current player count, and average rating, to support rapid decision-making.
Key features include:

\begin{itemize}
	
	\item \textbf{Filtering and Sorting:} Options to organize courts by proximity, rating, occupation or conditions.
	\item \textbf{Favourites:} A quick-access button to save frequently visited courts, a heart icon was chosen over a star for favourites to align with younger users' social media mental models, a choice validated during user testing.
	\item \textbf{Community Contribution:} A "plus" button near the search bar allows users to submit new courts, ensuring the database remains up-to-date through crowd sourcing.
	
\end{itemize}


Users can toggle between a standard list for quick scanning and an interactive map to visualize court locations spatially (Figure \ref{fig:courtsMapMU}). This map view provides spatial visualization of courts while maintaining the same informative list cards for consistency in data presentation.

\subsubsection{Court Details and Functional Hub}

Upon selecting a specific court, the user is directed to the "Court Details" screen (Figure \ref{fig:cdGPCMU}). This is divided into two sections to optimize information hierarchy. 
The upper section contains persistent high-level data, such as court imagery, information about the court, and the "Check-in" action. 

The lower section houses the tabbed interface. Given the high density of features, including game schedules, reviews, and rankings, a tabbed navigation system was implemented within this view in order to prevent mental overload associated with excessive forward-and-backward screen transitions.
This architecture allows users to access complex data subsets without losing the primary context of the selected court. 
The hub is organized into five functional tabs:	

\begin{itemize}
	
	\item \textbf{Game Tab:} Displays upcoming and past games, and provides the entry point to the "Create Game" flow.
	\item \textbf{Calendar Tab:} Shows the number of games per day and "player intentions." Users can log their own intent to play, helping others judge how busy the court will be.
	\item \textbf{Players Tab:} Lists users currently checked-in at the court, leading to individual "Player Profiles."
	\item \textbf{Rank Tab:} Displays the local leader board for that specific court.
	\item \textbf{Reviews Tab:} Allows users to read feedback on court conditions and contribute their own ratings.
	
\end{itemize}


The initial tabs include the Games, Players, and Classification views seen in Figure \ref{fig:cdGPCMU}. To maintain design consistency across the application, the game and player cards utilized here are identical to those found in the dedicated Games and Players modules (detailed in Sections \ref{subsec:GamesModule} and \ref{subsec:PlayersModule}), while the classification tab, shows the different leader boards across the diverse game formats.

\subsection{Games Module}
\label{subsec:rp-games}

The ability for users to organize and coordinate games is a core requirement of the application. 
Consequently, the Games module focuses on three primary functional pillars: discovery, creation, and comprehensive game management.

\subsubsection{Game Discovery and Selection}

Building upon the games list found within the Courts module, a dedicated "Games" screen was designed to provide a centralized marketplace for user-generated games. As illustrated in Figure \ref{fig:gamesListMU}, this interface allows users to browse scheduled matches across all courts. Key features integrated into this view include:

\begin{itemize}
	\item \textbf{Filtering and Sorting:} Mechanisms to organize results by geographic proximity, date, or game format (e.g., 3x3, 5x5).
	\item \textbf{Quick Entry:} A direct "Join" button allows users to register for a game without navigating away from the list, streamlining the interaction for frequent users.
	\item \textbf{Search by Invitation:} A specialized search field where users can input a unique code to join private or specific games directly.
	\item \textbf{Global Creation:} A prominent "Create Game" button serves as the primary entry point for the customization workflow.
\end{itemize}

\subsection{Players Module}
\label{subsec:rp-players}

The Players module facilitates social connectivity and identity management within the application. 
It is designed to serve a dual purpose: allowing users to manage their personal identity and enabling the discovery of potential team-mates or opponents.

\subsubsection{Player Profile and Personal Dashboard}

A central requirement for the application is a comprehensive profile system that supports both social networking and individual performance tracking.
The Profile screen, illustrated in Figure \ref{fig:profile1MU}, acts as a centralized repository for a user's statistics, social connections, and facility preferences.

Following the visual language established in the Courts module, the interface utilizes a 2 main components layout. The "Hero" section provides a visual anchor, displaying the user's profile image alongside critical biometric and stylistic metadata, such as age, height, and play style classification (e.g., relaxed, mixed, or intense).

The lower section of the profile employs the established tabbed navigation system to organize diverse data subsets without cluttering the interface:
\begin{itemize}
	\item \textbf{Statistics:} Provides quantitative data regarding games played and performance metrics (Figure \ref{fig:profileStatsMU}).
	\item \textbf{Games:} A chronological history of past and upcoming participation, utilizing the same standardized game-card component found in the Games module for design consistency (Figure \ref{fig:profileGamesMU}).
	\item \textbf{Friends:} A social directory of the user's established connections within the platform (Figure \ref{fig:profileFriendsMU}).
	\item \textbf{Teams:} Displays the various teams created through the platform to which the player belongs.
	\item \textbf{Courts:} A list of the user’s bookmarked or favourite courts (Figure \ref{fig:profileCourtsMU}).
	\item \textbf{Bio:} A qualitative space for a detailed biography, including history with official competitive organizations.
\end{itemize}

\subsection{Home Dashboard}
\label{subsec:rp-home}

The Home Dashboard serves as the application's primary landing surface and root destination. Its design centers on functional aggregation, surfacing prioritized data from the Courts, Games modules to provide users with immediate access to high-value features and personal schedules.

As illustrated in Figure \ref{fig:homeMU}, the dashboard acts as a centralized hub, integrating several key components:
\begin{itemize}
	\item \textbf{Personalized Weekly Calendar:} A condensed view showing the user's upcoming games and attendance intentions, allowing for rapid schedule verification upon app launch.
	\item \textbf{Game Discovery:} A list of active games available to join, reducing the navigation steps required to find a match.
	\item \textbf{Court Recommendations:} Summaries of local or favourite courts, providing a shortcut to the Courts module.
	\item \textbf{Create Game Button} A button to facilitating the app's core objective of match organization.
\end{itemize}

During the design phase, several architectural iterations were explored to balance information density with visual clarity. The layout presented in Figure \ref{fig:homeMU} was selected for initial user testing to evaluate whether this consolidated "overview" approach effectively minimized the cognitive load associated with navigating through separate modules for daily tasks.

\section{Evaluation I - Prototype Testing}
\label{sec:rp-eval1}

After completing the design phase and closing one cycle of the UCD process, the first evaluation was conducted. 
This user testing phase aimed to verify, through an initial prototype, whether the requirements gathered during user research were adequately reflected in the design. 
Three sessions were carried out: two individual sessions, one with a male participant over 40 years old and another with a participant between 25 and 30 years old, and one group session with three male participants between 20 and 25 years old.

This section presents the evaluation methods used, the results obtained, and the impact of these findings on the prioritisation of the Scrum User Stories.


\subsection{Method}
\label{subsec:rp-eval1-method}

Two complementary methods were used in this evaluation phase: task-based think-aloud testing and feature prioritisation.



\subsubsection{Tasked Based}
\label{subsubsec:method-task}

During this part of the evaluation, participants were given a mobile phone and asked to interact with the Figma interactive prototype. 
The prototype allowed navigation across the mockups through clickable buttons, enabling users to move between screens and explore the proposed flows.

As this was an early prototype, no information could be edited by the participants. 
All input fields were empty or prefilled, and any court or game selected redirected the user to the same corresponding mock screen. 
Participants were given nine tasks and were encouraged to verbalise their thoughts throughout the interaction, including what they were doing, what they expected to happen, what they liked or disliked, and any difficulties they encountered. 
At the end of each task, they were asked to report how easy the task had been, how frustrated they had felt, and what had caused that reaction.
They were also encouraged to provide additional comments on the user interface, user experience, and the proposed features, including aspects they considered useful, unnecessary, or missing.

The tasks were the following:

\begin{enumerate}
	\item Create an account;
	\item Verify existing games in a nearby court;
	\item Create a new team;
	\item See how many players are going to be in the court on a specific date and time;
	\item See the players that participated in the last game the user won;
	\item Create a new game;
	\item Search courts on the map;
	\item See the user's next game;
	\item See the classification table of a court.
\end{enumerate}

From the interviewer’s perspective, notes were taken whenever a participant provided significant feedback or comments while performing a task. At the end of each task, the interviewer also checked whether the participant had completed it successfully, whether help was required, and how difficult the task had been.

\subsubsection{Feature Prioritization}
\label{subsec:method-featPri}

The second method, which was particularly valuable in this phase, consisted of an online whiteboard containing the User Stories represented as sticky notes, together with an empty table divided into five categories: \textit{Essential}, \textit{Important}, \textit{Nice to Have}, \textit{Not Needed}, and \textit{Missing}.

Participants were asked to classify each User Story into one of these categories and to add complementary features in the \textit{Missing} column when relevant. They were also encouraged to annotate each User Story with comments that could help refine the requirements according to user needs and preferences.

This method made it possible to identify which features should be prioritised for implementation, which could be deferred if time allowed, and which were considered unnecessary. 
It also allowed the identification of features that had not been considered during the initial requirements gathering. 
To rank the User Stories, numerical values were assigned to each category, from 3 for \textit{Essential} to 0 for \textit{Not Needed}. These values were then multiplied by the number of users who selected each category for a given User Story, and the results were summed to obtain a final score. 
Given that five participants took part, the maximum possible score was 15 points.

\subsection{Results}
\label{subsec:rp-eval1-results}

\subsubsection{Tasked Based}


In the task-based tests, participants successfully completed all the assigned tasks.
The only task that consistently caused difficulty was creating a new team, mainly because the button to create it was located too deeply in the navigation structure.

The remaining tasks were completed without assistance or hesitation by most participants.
Throughout the session, additional comments were gathered, although several of the most relevant observations were repeated later during the feature prioritisation exercise.

\subsubsection{Feature Prioritization}

This method proved more informative than the previous one for this phase, as it made it possible to define a roadmap for the implementation of the platform’s features.
Appendix~\ref{appendix:evalI} contains the five boards completed by the participants. 
Based on those boards, Table~\ref{tab:US-ranked-tab} was created to calculate and rank the User Stories.

Five participants classified each User Story. 
The most highly valued features were game creation, authentication, and profile management, which were considered essential for any application involving social interaction and coordination.
Game invitations and court search functionality were also rated as high-priority features. 
On the other hand, participants assigned low value to spectator mode, court check-in, and game start functionality.

Beyond the classification of the User Stories, participants also provided qualitative feedback on feature relevance and design concerns. 
The main themes emerging from these comments are summarised below.

\paragraph{Game Flow and Automation}
Several participants suggested that manual game lifecycle actions should be automated where possible.In particular, starting and ending a game were considered unnecessary sources of friction if they could be triggered automatically when attendance was complete or when players checked in (\texttt{PLY-17}, \texttt{PLY-18}).

\paragraph{Open vs.\ Private Game Logic}
Participants questioned the usefulness of join requests in public games, arguing that open games should allow direct registration to encourage participation, while private invitation-only games could serve as the controlled alternative (\texttt{PLY-14}, \texttt{PLY-15}).

\paragraph{Court Data Reliability}
Concerns were raised about the reliability of user-submitted court occupancy data and the risk of false or unsafe court submissions. Participants suggested introducing a validation or moderation layer for new court additions, including image-based verification and administrative approval workflows (\texttt{PLY-27}, \texttt{PLY-28}).


\paragraph{Ranking and Classification System}
Participants identified the ranking system as an important feature but also pointed out several design challenges. These included uncertainty about whether rankings should be maintained per court or globally, the absence of a clearly defined algorithm that could be explained to users, and concerns about data integrity. One participant suggested that rankings should be verified or computed by the application rather than derived from potentially unreliable self-reported game results (\texttt{PLY-20}).

\paragraph{UI and Navigation Density}
Unsolicited design feedback indicated that the prototype felt information-dense, with too many actionable elements displayed on each screen. Participants recommended reducing the number of clicks required to complete common actions and keeping users on the same page whenever possible through internal tabs and collapsible sections. 


\paragraph{Missing Features Identified}
Participants also identified two missing capabilities that were not part of the original backlog: a gamification system, including badges, achievements, and court-based leaderboards, and a court activity indicator based on historical game frequency rather than live headcount.

\subsection{Impact on Backlog}
\label{subsec:rp-eval1-impact}

The feature prioritisation scores served as the main input for sprint allocation and for defining the scope and organisation of Phase 2 development. 
However, the final backlog organisation was also influenced by two additional factors: technical dependencies between features and implementation effort relative to expected value.\\

Technical dependencies established the implementation foundation regardless of priority score. Authentication and basic profile management had to be implemented before any other feature, as they form the basis of a social platform and are necessary for enforcing role-based database rules.
Courts also had to exist before games could be created, since games need to be associated with a court. Likewise, games had to exist before game details, invitations, or notifications could become meaningful. 
These constraints defined the implementation order across the sprints.

Court Check-In (\texttt{PLY-26}), despite obtaining a low score in the prioritisation exercise (3 points), was included in the MVP scope because it provides the data source for Court Occupancy Display (\texttt{PLY-27}).
Without check-in data, live occupancy indicators would not have meaningful input, and the feature would not be functional.
Although this dependency made Court Check-In technically necessary, participant feedback also suggested that check-in data alone would not be sufficiently reliable, indicating that the feature would require further iterations.

Similarly, Add Court to Favourite List (\texttt{PLY-45}, 8 points) was implemented earlier than its score alone would suggest because it reused data structures already introduced in the Courts module, which made it relatively inexpensive to develop while contributing to a more complete user experience.

The MVP scope was therefore defined as the minimum set of features required to validate the core value proposition of the platform: helping players find courts and organise games. Based on the prioritisation results and the identified dependencies, the following features were included in Phase 2:

\begin{itemize}
	\item User authentication and profile management (\texttt{PLY-37}, \texttt{PLY-38}), rated essential by all five participants.
	\item Game creation, browsing, and invitation management (\texttt{PLY-10}, \texttt{PLY-12}, \texttt{PLY-13}, \texttt{PLY-41}), core coordination features.
	\item Court search, details, and submit new courts (\texttt{PLY-42}, \texttt{PLY-46}, \texttt{PLY-40}, PLY-28), required to provide the context needed for game creation.
	\item Announce playing intentions (\texttt{PLY-30}), rated highly and directly addresses the coordination problem.
	\item Favourite courts (\texttt{PLY-45}), low-priority from a user perspective, but technically lightweight and therefore implemented alongside court details.
	\item Basic court occupancy (\texttt{PLY-27}) and court check-in (\texttt{PLY-26}), the latter scored low, but it is required for occupancy data to exist.
	\item Home Page and Navigation Bar, this User Stories were not in the task, but essential.
\end{itemize}

The following features were explicitly assigned to Phase 3 or removed from scope due to their complexity and lower immediate relevance, even if some of them received moderate scores in the prioritisation exercise. They were not essential for validating the platform’s core value proposition in the initial release:

\begin{itemize}
	\item Search Courts with a Map (\texttt{PLY-46}),  list-based search already provides a simpler first implementation.
	\item Team management (\texttt{PLY-22}) and player ranking (\texttt{PLY-20}).
	\item Friends system (\texttt{PLY-32}, \texttt{PLY-33}) and player search (\texttt{PLY-36}).
	\item Game chat (\texttt{PLY-34}).
	\item Push notifications (\texttt{PLY-11}, in Phase 2 \textit{in-app} notifications were implemented.
	\item Spontaneous game creation (\texttt{PLY-25}), merged into the standard game creation flow, based on participant feedback suggesting that the distinction was unnecessary.
	\item Tournament creation (\texttt{PLY-24}) and spectator mode (\texttt{PLY-19}), among the lowest-priority features and therefore deferred to future work.
\end{itemize}

The qualitative feedback also indicated that the classification system should be revised before implementation and that an administrator role should be introduced to validate and approve new courts more easily. This suggestion is particularly relevant for ensuring that the platform can grow with a wider and more reliable court database, which is essential for reaching more players and increasing the platform’s usefulness.

\subsection{Authentication and On boarding}

User authentication follows a three-step onboarding sequence: account creation, email verification, and profile completion.
The decision to separate account creation from profile setup was deliberate: it allowed the authentication layer to remain fully delegated to Supabase Auth while the profile data was handled by the application-level \texttt{players} table, as described in Section~\ref{subsec:arch-schema}.
This boundary also made it possible to enforce profile completeness independently of authentication state, so that a user with a verified account but an incomplete profile is redirected to the profile screen on every login until the required fields are submitted.

The registration form implements live password validation with a visual strength indicator and field-level error messages that appear inline rather than in a toast, reducing the number of re-submission cycles observed in informal testing.
To protect the backend from brute-force abuse, Supabase's rate-limiting policy was surfaced in the UI as a countdown timer that prevents re-attempts for five minutes after five failed authentication requests.
Session tokens are persisted in platform secure storage (Android Keystore / iOS Keychain, via the \texttt{expect}/\texttt{actual} mechanism described in Section~\ref{subsec:expect-actual}), so returning users are authenticated immediately on launch without re-entering credentials. 

A known iOS behaviour is that Keychain data persists across app uninstalls, meaning sensitive credentials stored during a previous installation remain accessible upon reinstall. This can lead to stale or mismatched authentication state, causing subtle but critical bugs on first launch. To address this, a first-launch Keychain cleanup routine was implemented, exploiting the asymmetry between \texttt{NSUserDefaults}, which is wiped on uninstall, and the Keychain, which is not. On the first launch following a fresh install, all generic password entries associated with the application's service name are deleted via the Security framework's \texttt{SecItemDelete} API. A persistent flag is then written to \texttt{NSUserDefaults} to ensure the cleanup runs exactly once, leaving subsequent launches and over-the-air updates entirely unaffected. The Listing \ref{lst:keychain-cleanup} shows the implementation,

\begin{lstlisting}[language=java, caption={iOS Keychain cleanup on fresh install}, label={lst:keychain-cleanup}]
	private fun clearStaleIosKeychainOnFreshInstall() {
		val defaults = NSUserDefaults.standardUserDefaults
		if (defaults.boolForKey(IOS_FIRST_LAUNCH_DONE_FLAG)) return
		
		val query = CFDictionaryCreateMutable(null, 2, null, null)
		CFDictionaryAddValue(query, kSecClass, kSecClassGenericPassword)
		CFDictionaryAddValue(query, kSecAttrService,
		CFBridgingRetain(IOS_KEYCHAIN_SERVICE_NAME))
		
		SecItemDelete(query)
		defaults.setBool(true, IOS_FIRST_LAUNCH_DONE_FLAG)
	}
\end{lstlisting}

A non-trivial implementation challenge arose from the email verification and password-reset flows, both of which rely on Supabase sending the user a link that must reopen the application rather than a browser.
This required configuring a deep link entry point on each platform.
On Android, an \texttt{intent-filter} was added to the \texttt{AndroidManifest.xml} inside the \texttt{androidApp} module, declaring the application as the handler for the Supabase redirect URL scheme:

\begin{lstlisting}[language=xml, caption={Deep link intent filter — androidApp/AndroidManifest.xml}]
	<intent-filter android:autoVerify="true">
	<action android:name="android.intent.action.VIEW" />
	<category android:name="android.intent.category.DEFAULT" />
	<category android:name="android.intent.category.BROWSABLE" />
	<data
	android:scheme="https"
	android:host="theplayground.pt"
	android:pathPrefix="/auth/v1/callback" />
	</intent-filter>
\end{lstlisting}

When the user taps the link in the email, Android resolves the intent to the application and delivers it to \texttt{MainActivity}.
The activity forwards the intent to the Supabase client, which extracts the token from the URL fragment and exchanges it for a valid session:

\begin{lstlisting}[language=java, caption={Deep link handling — androidApp/MainActivity.kt}]
	override fun onCreate(savedInstanceState: Bundle?) {
		super.onCreate(savedInstanceState)
		supabase.handleDeeplinks(intent)
	}
	
	override fun onNewIntent(intent: Intent) {
		super.onNewIntent(intent)
		supabase.handleDeeplinks(intent)
	}
\end{lstlisting}

On iOS, the equivalent entry point is the SwiftUI \texttt{onOpenURL} modifier, which receives the URL when the system routes a Universal Link or custom scheme to the application:

\begin{lstlisting}[language=swift, caption={Deep link handling — iosApp/iOSApp.swift}]
	ContentView()
	.onOpenURL { url in
		Task {
			try? await supabase.auth.session(from: url)
		}
	}
\end{lstlisting}

\subsection{Courts Module}


The Courts module exposes two primary surfaces in Phase~2: the court list and the court detail screen.
A map view was planned as part of the Courts module (Section~\ref{subsec:rp-courts}) but was deferred to Phase~3, the list-based interface was prioritised first as it provides the same discovery capability with lower implementation complexity, in line with the scope decision recorded in Section~\ref{subsec:rp-eval1-impact}.

The court list is backed by a \texttt{CourtsViewModel} that holds the full observable state, including the current court set, active filters, and the user's location.
The list loads all the courts from the district, and the court list item shows the following information:
\begin{itemize}
	\item Distance to the user's location (if location service is active);
	\item Current number of players in the court (check-in);
	\item Number of baskets;
	\item Number of full or half court;
	\item Heart, for the player to add or remove courts from favourites.
\end{itemize}

Filtering and sorting are applied within the ViewModel without triggering a new remote fetch, keeping interactions responsive and reducing unnecessary network calls.
If the location is active the courts are filter by closest, besides, the user can sort by most baskets or most players in the court.
In terms of filters, the user can filter by the courts characteristics (number of baskets, net type) and a range of number of player in the court.
Users can also search by name, showing new results using a de-bounce time, so it only makes a call when after a number of milliseconds of changing the query so no excessive calls are made in every keystroke.
To keep the list updating, there is an observable function to keep track of the user location and show the courts closer at all time.
The court list item in the list, shown in the Figure \ref{fig:courtsListP2}, is reused in the users profile screen in the favourite courts.

\subsection{Games Module}

The Games module covers the full lifecycle of a pickup game: creation, browsing, player management, and completion.

User can create games from several places, when creating a game, instead of a two step screen, the user scrolls down to have all the information all in one screen without moving around to edit details.
The Figure 
%The court selector in step one prioritises the user's favourite and recently visited courts at the top of the list, reducing the navigation required for the common case.
The second step configures team formation (manual or random), a ranked match flag, and, for private games, the initial player invitations.
When manual formation is selected, the host can assign invitees to a specific team at creation time; when random is selected, teams are distributed automatically by the \texttt{assign\_random\_game\_teams} server-side function when the game starts.

The join flow differs between public and private games.
Public games expose a direct join button that calls \texttt{join\_public\_game} immediately while private games require a join request, which the host must explicitly accept or reject from the game detail screen.
Both flows persist their state in the database (\texttt{game\_join\_requests} and \texttt{game\_invites}) so that the full history of pending, accepted, and declined actions remains visible in the interface and does not disappear on app restart.

Team management is performed through a contextual menu on each player card in the game roster.
%A three-dot icon opens options to view the player's profile, move the player to the opposite team (unilateral), or swap the player with a specific player on the other team.
Clinking on player, it redirects to the players profile, while pressing and moving can swap the player to another team, and dropping on another player swaps them.
The server-side \texttt{move\_player\_to\_team} and \texttt{swap\_players\_in\_game} functions enforce these operations atomically, preventing partially completed team changes.

The game lifecycle is represented by the \texttt{game\_status} enum (\texttt{ACTIVE} $\rightarrow$ \texttt{LIVE} $\rightarrow$ \texttt{COMPLETED}), and the game detail screen adapts its available actions to the current state.
Only the host can start and end the game, and assign a new host.
Ending the game requires the host to enter a final score, with the server validating that the score is non-null for both teams, and non-tied before persisting the \texttt{COMPLETED} state.
Once completed, the game's result is available in the game history visible on each player's profile and in the court's game tab.

\subsubsection{Home Dashboard and Notification}

The Home dashboard aggregates data from multiple modules into a single entry surface.
It is composed of four sections: a personal weekly calendar strip (the same as the one used in the court) showing the user's upcoming games, a next-game card showing the next scheduled game with its location and roster status, a list of active nearby games available to join,  and a list of courts organised in a way that first shows the latest checked in courts, than the favourites and then by location.

The "nearby" sections are derived from the user's current location.
The application requests location permission on first launch; if granted, courts and games are filtered and sorted by distance.

The Home screen is also the primary entry point for the notification panel, accessible via a bell icon in the top bar.
The panel presents all notifications grouped by category (games, general) and allows the user to navigate directly to the relevant game from the notification item.

In-app notifications are persisted in the \texttt{in\_app\_notifications} table and pushed to the client via a Realtime subscription filtered to \texttt{recipient\_id = auth.uid()}.
When a notification arrives while the user is in the application, a banner is displayed regardless of which screen is active.
A notification badge on the Home screen icon indicates that has notifications to read.

\subsubsection{Cross-Cutting Implementation Concerns}


\paragraph{Internationalisation.}
The application supports Portuguese and English, with string resources managed through Compose Multiplatform's \texttt{composeResources} system using \texttt{values/} and \texttt{values-pt/} directories.
All user-facing strings are externalised, no hardcoded text appears in composable functions.
The language is resolved at runtime from the device's system language setting.

\paragraph{Error Handling.}
Every repository method returns a sealed \texttt{Result} type rather than throwing exceptions.
With this, when a request is made, there is always two implementations, an \texttt{onSuccess} and a \texttt{onFailure}. 
ViewModels map domain errors to localised \texttt{UiState} fields, and the UI renders error states using standardised components (inline field errors, dismissible snackbars, full-screen error views with retry actions) depending on the severity and context of the failure.
Network errors always show cached data when available, satisfying NFR-03 (graceful degradation).

\paragraph{Form Validation.}
Form inputs use a validation pipeline that evaluates rules on each keystroke, displaying inline error messages as the user types rather than on submission.
Required fields are marked visually, and the submit action is disabled until all required fields pass validation.
